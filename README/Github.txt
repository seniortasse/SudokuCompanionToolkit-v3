1. Core principle

Never push blindly.

Use this order every time:

Inspect → Clean → Test → Stage intentionally → Commit → Push branch → Verify CI → Merge/tag/release when stable

For small safe changes, pushing directly to main can be acceptable. For large platform updates, new architecture, publishing engine changes, or many changed files, use a feature branch and pull request.

2. Recommended branch model

Use three kinds of Git history markers:

main

Stable working line. This should always be buildable or at least not knowingly broken.

work/... or feature/... branches

Use these for large development batches.

Examples:

work/publishing-platform-sync-2026-05
feature/step-solution-packages
feature/app-catalog-export
fix/android-ci-build
version tags

Use tags when a meaningful milestone is pushed.

Examples:

v0.7.0
v0.8.0
v0.9.0
v1.0.0

Suggested interpretation:

v0.7.0 — Publishing platform foundation + expanded app architecture
v0.8.0 — App catalog + step-solution packages stable
v0.9.0 — Voice solving loop and grid correction stable
v1.0.0 — First public-ready Sudoku Companion release

4. Files that should normally NOT be committed

Avoid committing generated outputs, local caches, private credentials, heavy datasets, and build artifacts.

Usually do not commit:

.venv/
.gradle/
build/
.cxx/
local.properties
*.jks
*.keystore
*.apk
*.aab
runs/
datasets/
exports/
models/
debug/
*.zip
*.png, except Android app resource images explicitly allowed by .gitignore
large generated PDFs

If a file is generated and reproducible, prefer to ignore it and document the command that generates it.

If a file is a large source asset that truly belongs in the repo, consider Git LFS or a separate artifact storage strategy.

5. One-time safety checks before a major push

Run from repo root:

git status --short
git branch -vv
git remote -v
git fetch origin
git log --oneline --decorate --graph --max-count=20 --all

Check ignored files:

git status --ignored --short

Check whether large files are about to be committed:

git ls-files | ForEach-Object {
  if (Test-Path $_) {
    $item = Get-Item $_
    if (-not $item.PSIsContainer -and $item.Length -gt 50MB) {
      [PSCustomObject]@{ SizeMB = [math]::Round($item.Length / 1MB, 2); Path = $_ }
    }
  }
} | Sort-Object SizeMB -Descending

Check untracked large files:

Get-ChildItem -Recurse -File -Force |
  Where-Object { $_.FullName -notmatch '\\.git\\' -and $_.Length -gt 50MB } |
  Select-Object @{Name='SizeMB';Expression={[math]::Round($_.Length / 1MB, 2)}}, FullName |
  Sort-Object SizeMB -Descending

Search for possible secrets before staging:

git grep -n "OPENAI_API_KEY\|AZURE\|SECRET\|PASSWORD\|TOKEN\|PRIVATE KEY" -- .

For untracked files, also search with PowerShell if needed:

Get-ChildItem -Recurse -File -Force |
  Where-Object { $_.FullName -notmatch '\\.git\\|\\.venv\\|\\build\\|\\.gradle\\' } |
  Select-String -Pattern "OPENAI_API_KEY|AZURE|SECRET|PASSWORD|TOKEN|PRIVATE KEY" -CaseSensitive:$false

10. Emergency recovery commands

See what changed:

git status --short
git diff --stat
git diff --cached --stat

Unstage everything, keep local files:

git restore --staged .

Discard changes to one file:

git restore path/to/file

Discard all uncommitted tracked-file changes:

git restore .

Remove untracked files preview only:

git clean -nd

Actually remove untracked files:

git clean -fd

Be very careful with git clean -fd; it deletes untracked files locally.

See recent commits:

git log --oneline --decorate --graph --max-count=20

Return to previous branch:

git switch -
